# üóìÔ∏è Meet in the Middle (Backend)

Meet in the Middle is a collaborative meeting scheduling and coordination platform.  
It helps users find optimal meeting times, share locations, manage invites, and integrate with Google Calendar and Stripe for premium subscriptions.

This repository contains the backend API built with Node.js, Express, and MongoDB.

# Meeting Functionalities

## üü¢ `createMeeting`

- Validates the incoming request body using **Joi schema** (title, time, participants, etc.).
- Starts a **MongoDB transaction** with `mongoose.startSession()` to ensure atomicity.
- Fetches the **creator user** from DB, and if not found ‚Üí unauthorized.
- Creates a **new meeting document** with title, description, creator, and timings.
- Generates a **meeting link** using `process.env.FRONTEND_URL`.
- Prepares **participants list**:
    - Adds participants excluding the creator.
    - Ensures if participant email exists, links to existing user.
    - Creator is added as an "Accepted" participant with location.
- Inserts all participants into DB (`insertMany`) and links them to the meeting.
- Commits the transaction, emits a `meetingCreated` event, and responds with success.

---

## üü¢ `getMeetings`

- Extracts logged-in user‚Äôs email.
- Applies pagination (`pageNo`, `items`) to fetch data in chunks.
- Finds all `Participant` entries where the user is a participant.
- Populates linked **meeting data, creator details, and participant info**.
- If no meetings found, returns `"No Meetings found"`.
- Otherwise, returns all meetings with populated data.

---

## üü¢ `getPendingMeetings`

- Gets logged-in user‚Äôs email and pagination.
- Finds **pending invitations** where status = `"Pending"`.
- Populates meeting + creator + participants info.
- Filters out duplicate meetings using `Map`.
- Returns a **formatted list** (id, title, host name, description, people count, date, time).
- Helps user quickly see which meetings are still **awaiting their response**.

---

## üü¢ `deleteMeeting`

- Takes `meetingId` from request params.
- Checks if meeting exists, else returns "Meeting not found".
- Deletes the **meeting document** and its participants.
- Loops through participants and **sends notifications** about cancellation.
- Builds **cancellation email HTML** and sends via `sendMeetingInvitationMail`.
- Responds with `"Meetings deleted successfully!"`.

---

## üü¢ `getMeetingById`

- Takes `meetingId` from params.
- Finds meeting by ID and **populates creator + participants**.
- If not found, returns `"Meeting not found"`.
- Otherwise, returns meeting details.

---

## üü¢ `editMeetingById`

- Finds meeting by `meetingId`.
- Updates **title and description** from request body.
- Saves updated meeting.
- Prepares **new invitation email HTML** with updated info.
- (Sending mail is currently commented out.)
- Responds with updated meeting.

---

## üü¢ `acceptMeeting`

- Extracts `meetingId` + location from request body, and logged-in user‚Äôs email.
- Finds the participant for this meeting.
- Prevents the **creator from accepting their own meeting**.
- If participant exists, sets status = `"Accepted"` and saves location.
- Notifies the creator that someone **accepted their invitation**.
- Responds with success message.

---

## üü¢ `rejectMeeting`

- Similar to `acceptMeeting`, but sets status = `"Rejected"`.
- Prevents the **creator from rejecting their own meeting**.
- Notifies the creator that participant **rejected** the meeting.
- Responds with success.

---

## üü¢ `conflicts`

- Finds the current meeting by `meetingId`.
- Fetches all participations for logged-in user (except Rejected ones).
- Compares times to check if **current meeting overlaps with others**.
- Returns conflicts (list of overlapping meetings).
- If none, says `"No conflicts found"`.

---

## üü¢ `dashboardStats`

- Builds user dashboard statistics:
    - Total meetings.
    - Upcoming meetings.
    - Pending invitations.
    - Meetings in current week.
    - Average participants per meeting.
    - Success rate (Accepted √∑ Total).
- Uses filters and calculations on `Participant` + `Meeting`.
- Returns a neat stats object.

---

## üü¢ `upcomingMeetings`

- Fetches participations of user.
- Filters only meetings with **future scheduledAt** date.
- Returns upcoming meetings with full details.

---

## üü¢ `recentActivity`

- Gets user‚Äôs recent activity:
    - Meetings created by user.
    - Participant actions (accept/reject) by user.
- Maps into activity objects (`type`, `user`, `target`, `timestamp`).
- Merges and sorts by latest timestamp.
- Returns a timeline-style activity feed.

---

## üü¢ `scheduleMeetingReminder`

- Takes `meetingId`.
- Finds meeting and participants.
- Filters participants with `status="Accepted"` and reminders enabled.
- Schedules **email reminders** at:
    - 1 day before.
    - 3 hours before.
- Uses `node-schedule` for scheduling.

---

## üü¢ `confirmationRemainder`

- Similar to reminders but for `"Pending"` participants.
- Calls `scheduleConfirmationRemainder` utility to send confirmation reminders.

---

## üü¢ `calculateEquidistantPoint`

- Finds meeting and participants with Accepted + location data.
- If at least 2 participants exist ‚Üí calculates **average latitude and longitude**.
- Returns the **midpoint (equidistant point)**.
- Useful for deciding a **central meeting spot**.

---

## üü¢ `acceptedParticipantsLocations`

- Fetches all Accepted participants with valid locations.
- Maps each into `{ name, email, lat, lng, placeName }`.
- Returns list of locations (for maps).

---

## üü¢ `nearByPlaces`

- Finds meeting, gets Accepted participants with location.
- Calculates equidistant point.
- Calls **Overpass API** (or Google Places) to fetch nearby places (`type=restaurant`, etc.).
- Returns list of nearby places (for meeting venue suggestions).

## üîπ Function: `sendOTP`

üëâ **Purpose:** Generate a one-time password (OTP), save/update it in the database, and send it to the user‚Äôs email.

### Step-by-step explanation:

- `const email = req.body.email;` ‚Üí Get the user‚Äôs email from the request body.
- `const otp = crypto.randomInt(100000, 999999).toString();` ‚Üí Generate a secure random 6-digit OTP (better than `Math.random()`).
- `await OtpModel.findOneAndUpdate(...)` ‚Üí Store OTP in the `OtpModel` collection.
    - If the email already exists ‚Üí update the OTP and `createdAt`.
    - If not ‚Üí create a new OTP record (thanks to `upsert: true`).
- `await sendVerificationEmail(email, otp);` ‚Üí Send the OTP to the user‚Äôs email.
- `res.status(201).json({ message: "OTP sent successfully" });` ‚Üí Respond back to the client confirming OTP was sent.
- If anything goes wrong ‚Üí `catch (error)` returns a 500 error with `"Failed to send OTP"`.

üëâ **In short:** This function **creates and sends a 6-digit OTP** to the user‚Äôs email, ensuring only one record per email is stored (update/insert).

---

## üîπ Function: `verifyOTP`

üëâ **Purpose:** Verify if the OTP is valid, create a new user, set their preferences, and send a welcome mail.

### Step-by-step explanation:

- `const { email, otp, password, name } = req.body;` ‚Üí Extract input values from request.
- `const otpData = await OtpModel.findOne({ email, otp });` ‚Üí Check if the OTP exists for the given email.
    - If not found ‚Üí return `"Invalid OTP"`.
- Compare `createdAt` with current time ‚Üí ensure OTP is not expired (`>6000 sec`).
    - If expired ‚Üí return `"OTP expired"`.
- `const existingUser = await UserModel.findOne({ email });` ‚Üí Make sure the user doesn‚Äôt already exist.
    - If exists ‚Üí return `"Email already exists"`.
- `const user = await UserModel.create({...})` ‚Üí Create a new user with name, email, and password.
- `await OtpModel.deleteOne({ email });` ‚Üí Remove OTP record after successful use (one-time only).
- `const userSettingsNew = await Preferences.create({ userId: user._id });` ‚Üí Create default preferences/settings for the new user.
- `await UserModel.findByIdAndUpdate(user._id, { settings: userSettingsNew._id });` ‚Üí Link the preferences to the user account.
- `sendWelComeMail(email)` ‚Üí Send a welcome email (done asynchronously, doesn‚Äôt block response).
- Finally ‚Üí `res.status(200).json({ message: "OTP verified successfully" });`.
- If any error happens in between ‚Üí `catch` block logs it and returns `"Failed to verify OTP"`.

## üîπ Function: `createCheckoutSession`

üëâ **Purpose:** Create a Stripe checkout session so the user can start a subscription (like Premium plan).

### Step-by-step explanation:

- `const session = await stripe.checkout.sessions.create({...})` ‚Üí Creates a new checkout session with Stripe.
- `customer_email: req.user.email` ‚Üí Uses the logged-in user‚Äôs email for Stripe billing.
- `payment_method_types: ["card"]` ‚Üí Allows card payments only.
- `mode: "subscription"` ‚Üí Tells Stripe this is a **recurring subscription** (not one-time).
- `line_items: [{ price: PLANS.premium.priceId, quantity: 1 }]` ‚Üí Defines what the user is buying:
    - Uses the `priceId` from your `PLANS` config (Premium plan).
    - Quantity is 1 (one subscription).
- `success_url` ‚Üí If payment is successful, Stripe will redirect to `/subscription-success`.
- `cancel_url` ‚Üí If user cancels checkout, redirect to `/subscription-cancel`.
- `billing_address_collection: "required"` ‚Üí Forces user to enter billing address.
- `metadata: { userId: req.user.id }` ‚Üí Attach the user‚Äôs ID for tracking (helpful in webhooks).
- `res.json({ url: session.url })` ‚Üí Send the session URL back so the frontend can redirect the user to Stripe checkout.
- If something fails ‚Üí `catch` block returns a `500` error with the error message.

üëâ **In short:** This function creates a Stripe checkout session where the logged-in user can subscribe to your Premium plan.

---

## üîπ Function: `createBillingPortalSession`

üëâ **Purpose:** Allow users to manage their subscription (update card, cancel plan, etc.) via Stripe‚Äôs billing portal.

### Step-by-step explanation:

- `const userId = req.user.id;` ‚Üí Get the currently logged-in user‚Äôs ID.
- `const subscription = await Subscription.findOne({ user: userId });` ‚Üí Find the subscription document in your DB for this user.
- `if (!subscription || !subscription.stripeCustomerId)` ‚Üí Check if this user has an active Stripe customer account.
    - If not ‚Üí return error `"No active Stripe customer found"`.
- `const portalSession = await stripe.billingPortal.sessions.create({...})` ‚Üí Create a new Stripe **billing portal session**.
- `customer: subscription.stripeCustomerId` ‚Üí Connect this session to the user‚Äôs Stripe account.
- `return_url: \`${process.env.FRONTEND_URL}/account``‚Üí After managing subscription, user comes back to`/account`.
- `res.json({ url: portalSession.url })` ‚Üí Send the billing portal URL to frontend.
- If anything fails ‚Üí log the error and return `500` with error message.

üëâ **In short:** This function lets a user open Stripe‚Äôs billing portal so they can manage their subscription details themselves.

## Models

### üîπ Meeting Model (`Meeting`)

- Stores meeting details like **title, description, creator, participants, and meeting link**.
- Has fields for **scheduled time, end time, and location suggestions**.
- Indexes created for fast lookups ‚Üí by **creator, participants, or date/time**.
- Used in dashboards, reminders, invites, and analytics.

---

### üîπ Notification Model (`Notification`)

- Stores **alerts/notifications** for users (meeting created, accepted, rejected, deleted).
- Linked to a **user** (receiver) and contains **message + extra data** (like meetingId).
- `isRead` flag tracks whether a notification is seen.
- Auto timestamps help in sorting notifications by time.

---

### üîπ OTP Model (`Otp`)

- Stores **email + OTP code** for verification/authentication.
- `createdAt` has TTL index ‚Üí OTP auto-expires after **10 minutes**.
- Indexed for **fast OTP validation** and fetching latest OTP.
- Used in **login, signup, and password reset flows**.

---

### üîπ Participant Model (`Participant`)

- Represents a **meeting invitee** (linked to a user and a meeting).
- Tracks invitee‚Äôs **status** (Pending/Accepted/Rejected) and **location**.
- Supports **conflict reasons** (if invitee can‚Äôt attend at certain times).
- Indexed for fast queries (by email, meeting, or status) and prevents duplicate invites.

---

### üîπ Preferences Model (`Preferences`)

- Stores a user‚Äôs **settings/preferences** (notifications, reminders, profile visibility).
- Linked **1-to-1 with a user** (unique `userId`).
- Default values provided for all toggles (like email notifications ON by default).
- Indexed on `userId` for **fast fetch/update** in settings.

---

### üîπ PushSubscription Model (`PushSubscription`)

- Stores **push notification subscription data** (email, endpoint, encryption keys).
- Used for **Web Push API** to send browser/device notifications.
- Links subscription to a **user‚Äôs email**.
- Auto timestamps ‚Üí track when a subscription was created.

---

### üîπ Subscription Model (`Subscription`)

- Stores **Stripe subscription details** (customerId, subscriptionId, plan).
- Tracks **plan type** (free/premium), billing status, and renewal date.
- Keeps subscription **status** (active, canceled, trialing, etc.).
- Linked to a user for billing and access control.

---

### üîπ User Model (`User`)

- Stores **user profile info** (name, email, password, avatar, bio, etc.).
- Handles **auth providers** (local, Google, Facebook) and login devices.
- Includes **password hashing + compare method** (using bcrypt).
- Indexed for fast lookups (by email, provider, IP) ‚Üí used in authentication.

## üîπ `createGoogleCalendarEvent`

*(Creates and pushes a meeting event into the user‚Äôs Google Calendar)*

- **Fetch user tokens** ‚Üí Finds the user in the DB by `userId` to check if they have Google OAuth tokens (needed to talk to Google Calendar).
- **Skip if no tokens** ‚Üí If tokens are missing, it logs and exits (so app doesn‚Äôt break for users without Google sync).
- **Setup OAuth2 client** ‚Üí Creates an OAuth2 client using Google client ID, secret, and callback URL from `.env`.
- **Attach tokens to client** ‚Üí Sets the access + refresh token from the user‚Äôs DB record into the OAuth client.
- **Initialize Calendar API** ‚Üí Prepares Google Calendar API (`google.calendar`) with the authorized client.
- **Define event details** ‚Üí Builds an event object with:
    - `summary` ‚Üí meeting title.
    - `description` ‚Üí custom or default description.
    - `start` and `end` times ‚Üí in ISO format, with timezone.
    - `attendees` ‚Üí meeting creator and participants‚Äô emails.
    - `reminders` ‚Üí email reminder (1 day before) + popup reminder (10 min before).
- **Insert event into Google Calendar** ‚Üí Calls `calendar.events.insert()` with `sendUpdates: "all"` to notify participants.
- **Return success or failure** ‚Üí Logs and returns event link on success, or logs an error if failed.

## üîπ `createAndSendNotification`

*(Creates a notification record in DB and sends it instantly via WebSocket)*

- **Check user preferences** ‚Üí Finds the user‚Äôs notification preferences in DB, specifically if `inAppNotification` is enabled.
- **Skip if disabled** ‚Üí If user has disabled in-app notifications, it logs and returns without sending.
- **Create notification in DB** ‚Üí Adds a new record in `Notification` collection with `userId`, `type`, `message`, and optional `data`.
- **Prepare real-time message** ‚Üí Constructs notification payload with `_id`, `type`, `message`, `data`, `isRead`, and `createdAt`.
- **Emit via Socket** ‚Üí Uses `sendNotification` (from socket config) to push the notification in real time to the user‚Äôs active sessions.
- **Log success** ‚Üí Prints in console for debugging (`Notification sent: ...`).
- **Handle errors** ‚Üí If DB save or socket fails, it catches and logs the error.
